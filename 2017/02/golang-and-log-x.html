How Does Go Compute a Logarithm

Feb 06, 2017

About a year ago, I was reading the Go source for computing log⁡(x)\log(x)log(x) and was very surprised by the implementation.1

Even three years into a PhD in Applied Math (i.e. numerics), I still managed to learn something by diving in and trying to understand what was going on.

(Almost-)Correctness

Really I just wanted a chance to share a neat error plot:

The high-level:

  • The computation log⁡(x)\log(x)log(x) can be reduced to computing a related function R(s)R(s)R(s) (where the value sss can be obtained from xxx on a computer in some straightforward way)
  • R(s)R(s)R(s) can be approximated by a function that we can actually compute with simple floating point operations: f(s)f(s)f(s)
  • The error R(s)−f(s)R(s) - f(s)R(s)−f(s) is "equioscillating", i.e. it minimizes the maximum error, a so-called minimax solution

If you're still interested in some mathematical details, stick around. If not, I hope you enjoyed the pretty picture.

Doing the Obvious

The "obvious" idea that comes to mind is to use a Taylor series approximation. But log⁡(x)\log(x)log(x) is not defined at x=0x = 0x=0, so a Taylor series of a related function is typically used2:

But, the implementation doesn't do the obvious thing. Instead it breaks down the argument into an exponent and a fractional part

x=2k(1+s1−s)x = 2^k \left(\frac{1 + s}{1 - s}\right)x=2k(1−s1+s​)

which transforms the computation to:

log⁡(x)=klog⁡(2)+[log⁡(1+s)−log⁡(1−s)].\log(x) = k \log(2) + \left[\log\left(1 + s\right) - \log\left(1 - s\right)\right].log(x)=klog(2)+[log(1+s)−log(1−s)].

The first part klog⁡(2)k \log(2)klog(2) can be handled in maximal precision for the limited set of integer values that kkk can be.

Not Go, Sun

You may have noted my typo above: I used R(z)R(z)R(z) when I should've used R(s)R(s)R(s). This typo is directly from the source; not my copy-paste error, but a copy-paste error in the original source.

The original authors aren't the Golang devs:

// The original C code, the long comment, and the constants
// below are from FreeBSD's /usr/src/lib/msun/src/e_log.c
// and came with this notice. The go code is a simpler
// version of the original C.

It seems they copied everything, even the typos. Another typo can be found when the description mentions "a special Reme algorithm" used to compute the coefficients LjL_jLj​.

This typo is a bit more egregious: it actually uses a Remez algorithm.

Remez Algorithm

This algorithm takes a polynomial approximation like
